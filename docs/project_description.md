# Описание проекта

## Общие сведения

Название проекта: Personal Agile Planner.

Вариант рейтинговой работы: "Разработка программы планировщика Agile для личного пользования".

Автор: Грузин Михаил Дмитриевич, группа о.ИЗДтс 23.1/Б3-24.

## Назначение разработки

Personal Agile Planner предназначен для личного планирования задач с использованием простых элементов Agile-подхода. Пользователь может вести список задач, назначать им приоритеты, отслеживать статусы выполнения и хранить данные локально в JSON-файле.

Разработка выполняется в рамках рейтинговой работы по дисциплине "Проектная деятельность в ИТ" и служит примером небольшого ИТ-проекта, для которого можно показать структуру работ, жизненный цикл информационной системы и применение системы контроля версий Git.

## Область применения

Программа может применяться студентом или индивидуальным пользователем для планирования учебных, рабочих и личных задач. Проект имеет учебный характер и используется для демонстрации структуры ИТ-проекта, этапов жизненного цикла информационной системы и настройки Git/GitHub.

## Пользователи

Основной пользователь - индивидуальный пользователь, который ведет собственный список задач. В рамках учебного проекта предполагается один уровень доступа: владелец локального списка задач.

Права доступа в учебной версии:

| Категория пользователя | Права |
| --- | --- |
| Владелец планировщика | добавление задач, просмотр списка задач, локальное хранение данных |
| Разработчик проекта | изменение исходного кода, тестирование, фиксация этапов в Git |

## Основные функции

- добавление задачи;
- просмотр задач;
- хранение задач в JSON-файле;
- работа со статусами backlog, todo, in_progress, done;
- указание приоритета задачи;
- подготовка структуры для дальнейшего тестирования.

## Функциональные требования

1. Программа должна запускаться из командной строки.
2. Программа должна поддерживать команду добавления задачи.
3. При создании задачи пользователь должен иметь возможность указать название, описание, приоритет и срок выполнения.
4. Программа должна сохранять задачи в локальном JSON-файле.
5. Программа должна выводить список сохраненных задач.
6. Каждая задача должна иметь один из статусов: backlog, todo, in_progress, done.
7. Проект должен содержать автоматический тест базовой операции добавления задачи.

## Нефункциональные требования

1. Проект должен использовать Python 3 и стандартную библиотеку.
2. Структура проекта должна быть понятной для учебного анализа.
3. Исходный код должен быть разделен на модули модели, хранения данных, бизнес-логики и CLI-интерфейса.
4. История разработки должна фиксироваться в Git-коммитах и ветках.

## Модель жизненного цикла

Для проекта используется каскадная модель с элементами итерационного уточнения:

1. Инициация проекта.
2. Сбор и описание требований.
3. Проектирование структуры программы.
4. Реализация функциональности.
5. Тестирование.
6. Подготовка релиза и публикация в GitHub после подтверждения.

## Связь веток Git с этапами жизненного цикла ИС

| Этап ЖЦ ИС | Ветка Git | Содержание работ |
| --- | --- | --- |
| Стабильная версия | `main` | хранение проверенной версии проекта |
| Разработка | `develop` | объединение текущих изменений до переноса в стабильную ветку |
| Инициация | `wbs/01-initiation` | определение темы, цели и назначения разработки |
| Требования | `wbs/02-requirements` | описание функциональных и нефункциональных требований |
| Проектирование | `wbs/03-design` | описание структуры папок, модулей и формата данных |
| Реализация | `wbs/04-implementation` | разработка моделей, хранения, логики и CLI |
| Тестирование | `wbs/05-testing` | проверка запуска программы и базовых тестов |
| Завершение | `wbs/06-release` | подготовка README, материалов отчета и публикации после подтверждения |

## Иерархическая структура работ

1. Инициация:
   - определение темы;
   - описание назначения программы;
   - выбор стека.
2. Требования:
   - описание функций;
   - определение пользовательских сценариев.
3. Проектирование:
   - структура папок;
   - модель задачи;
   - формат JSON-хранилища.
4. Реализация:
   - модуль модели;
   - модуль хранения;
   - модуль планировщика;
   - CLI-интерфейс.
5. Тестирование:
   - проверка создания задачи;
   - проверка сохранения и чтения данных.
6. Завершение:
   - подготовка README;
   - подключение удаленного репозитория;
   - подготовка материалов для отчета.

## Инструкция запуска

Просмотр справки:

```bash
python -m src.agile_planner.main --help
```

Добавление задачи:

```bash
python -m src.agile_planner.main add "Prepare report" --priority high --due-date 2026-05-10
```

Просмотр списка задач:

```bash
python -m src.agile_planner.main list
```

Запуск тестов:

```bash
python -m unittest discover -s tests
```
